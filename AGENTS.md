# Bridge 巡检工作台 AGENTS

本文件用于指导后续 AI Coding / 人类开发者持续实现一个 **Bridge-only 的 Electron 巡检工作台**。

该产品不是通用浏览器自动化平台，而是面向 **Windows 专用值守电脑** 的、围绕 **Midscene Chrome Extension Bridge 模式** 构建的任务工作台。请后续实现严格遵守本文档，不要回退到 CDP / Launch / Puppeteer 多模式产品思路。

---

## 1. 产品定位

### 1.1 目标产品

构建一个桌面 exe，运行在 Windows 上，提供以下能力：

1. 通过 Electron 提供任务工作台 UI
2. 通过 Midscene Chrome Extension 的 **Bridge 模式** 驱动桌面 Chrome
3. 在一台专用电脑上复用人工完成过复杂登录的 Chrome 会话
4. 定时巡检业务后台报表/指标页
5. 在发现异常时进行本地提醒和外部告警
6. 保存任务、历史结果、截图、报告和异常记录

### 1.2 明确边界

本产品 **不是**：

- 不是通用 RPA 平台
- 不是让用户在 Electron 内嵌 webview 中直接跑业务系统
- 不是自动登录器
- 不是 CDP/Bridge/Launch 三种模式并存的连接管理器
- 不是让普通办公用户在自己的日常浏览器上无感后台执行

本产品 **是**：

- 一套基于专用电脑、专用 Chrome 会话的 Bridge 自动化工作台

---

## 2. 核心场景与默认假设

### 2.1 核心场景

目标场景为：

- 用户可能有多个业务后台需要每隔 10 分钟巡检
- 系统无法改造，没有开放 API
- 登录链路复杂，无法通过自动化完成登录
- 用户可提供一台专用 Windows 电脑，外接大屏，长期保持 Chrome 登录态
- 自动化重点是“读取页面、提取数据、判断异常、发出提醒”

### 2.2 运行假设

后续实现默认基于以下前提：

1. 目标机器为 **Windows**
2. 已安装 **Google Chrome**
3. 已安装 **Midscene Chrome Extension**
4. Chrome 中已经人工完成目标系统登录
5. 该电脑主要用于自动化任务，不与日常办公电脑混用
6. Bridge 服务默认运行在本机回环地址

如果某个实现与这些前提冲突，应优先维护本文件中的产品边界，而不是为了“兼容更多模式”增加复杂度。

## 3. 架构原则

### 3.1 Bridge-only

这是本产品最重要的约束。

后续实现必须满足：

- 只支持 **Bridge 模式**
- 不重新引入 `launch` / `cdp` / `puppeteer` 用户可见选项
- 不为“未来可能支持 CDP”预留复杂 UI
- 不在运行时根据不同模式分叉出多套配置结构

允许的唯一例外是：内部保留极少量兼容迁移代码，用于读取老配置，但保存后必须收敛为 Bridge 专用结构。

### 3.2 Electron 只做工作台，不做目标浏览器

Electron 的职责：

- 任务配置
- 调度
- 运行状态展示
- 告警中心
- 报告查看
- 本地数据管理

Electron **不应**承担：

- 承载目标业务系统的登录态
- 作为真实自动化执行页面容器
- 复刻浏览器行为

真实自动化执行环境始终是 **桌面 Chrome + Midscene 扩展**。

### 3.3 专用值守电脑优先于通用办公电脑

产品设计优先服务于“专用值守机器”的稳定性，而不是兼容普通用户正在使用的日常办公浏览器。

因此：

- 定时任务优先使用 **Bridge `newTabWithUrl`**
- 手动调试时可支持 **`currentTab`**
- 不以“尽量不影响用户当前 tab”为最高目标，因为默认前提就是专机值守

### 3.4 监控巡检优先于通用脚本编排

第一优先级是：

- 稳定打开目标页面
- 等待页面可用
- 提取结构化数据
- 执行规则判断
- 保存证据
- 可靠告警

## 4. 连接策略设计

### 4.1 默认连接方式

定时任务默认采用：

- Bridge 模式
- `newTabWithUrl`
- 任务结束后允许关闭自动打开的新标签页

原因：

- 仍可复用已登录的 Chrome 会话
- 每次任务从相对可控的入口页开始
- 减少当前标签污染
- 更适合无人值守定时执行

### 4.2 手动调试方式

手动调试/录制/排障时允许：

- Bridge 模式
- `currentTab`

适用场景：

- 用户已手工打开正确页面
- 希望 AI 直接接管当前页
- 用于问题复现、调试提示词、校正提取规则

### 4.3 不做的事

后续实现中不要新增：

- “启动内置浏览器”
- “连接 CDP”
- “自动发现 Chrome 调试端口”
- “启动 Chromium”
- “通过 Puppeteer 管理浏览器生命周期”

## 5. 模型策略（非常重要）

### 5.1 基本判断

**需要截图理解。**

Midscene 的 UI 自动化默认依赖视觉路线。元素定位、任务执行、页面理解都与截图上下文密切相关。

因此不要将模型策略设计成：

- 一个视觉模型负责识别
- 一个纯文本模型负责规划

这不是当前 Midscene 的最佳实践。

### 5.2 正确认知

Midscene 中至少有三类模型角色：

1. **默认模型（必填）**
   - 负责元素定位
   - 负责未被独立配置的其他意图
   - 必须是适合 UI 自动化的视觉模型

2. **Planning 模型（可选）**
   - 负责 `ai` / `aiAct` 的任务规划
   - 即使是规划，也通常依赖 UI 上下文，不应假设为纯文本模型
   - 如需单独配置，应选择对 UI 交互理解较好的多模态模型

3. **Insight 模型（可选）**
   - 负责 `aiQuery` / `aiAsk` / `aiAssert` / `aiWaitFor` 等页面理解、数据提取与断言
   - 对巡检场景尤其重要

### 5.3 对本产品的建议

UI 中支持以下模型配置结构：

1. **默认执行模型（必填）**
2. **页面理解模型（可选，高级设置）**
3. **任务规划模型（可选，高级设置）**

默认行为：

- 未单独配置 Insight 时，`aiQuery/aiAssert` 回落到默认模型
- 未单独配置 Planning 时，`ai/aiAct` 回落到默认模型

### 5.4 配置保存建议

后续实现建议在应用中保存三组模型配置，但 UI 上分层展示：

- 简洁模式：只显示“默认执行模型”
- 高级模式：展开“任务规划模型”和“页面理解模型”

每组配置建议支持：

- `baseUrl`
- `apiKey`
- `modelName`
- `modelFamily`
- 可选自定义 headers / init config

内部映射建议遵循 Midscene 约定：

- `MIDSCENE_MODEL_*`
- `MIDSCENE_PLANNING_MODEL_*`
- `MIDSCENE_INSIGHT_MODEL_*`

### 5.5 对编码 Agent 的明确要求

后续 AI Coding 必须遵守：

- 不要把 Planning 实现成“纯文本 LLM 专用通道”
- 不要假设“规划不需要图片”
- 不要省略默认视觉模型配置
- 监控类任务优先使用 **`aiQuery + JS 规则判断`**
- `aiAssert` 适合做页面状态检查与粗粒度判断，不应承担所有数值判断逻辑

## 6. 任务执行策略

### 6.1 任务定义应偏结构化，而非纯 prompt

虽然底层仍可支持自然语言任务，但产品层的任务对象应尽量结构化。建议每个巡检任务至少包含：

- 任务名称
- 目标系统名称
- 入口 URL
- 运行模式（默认 `newTabWithUrl`，调试可 `currentTab`）
- 页面就绪条件
- 数据提取定义
- 规则定义
- 调度配置
- 告警配置
- 失败重试配置

### 6.2 页面就绪建议

进入页面后不要立即开始取数，优先做：

1. 导航到目标 URL
2. `aiWaitFor` 页面关键标题/关键卡片出现
3. 必要时二次刷新或等待数据加载
4. 再执行 `aiQuery`

### 6.3 取数策略

对于报表巡检，优先使用：

- `aiQuery<T>()` 提取结构化 JSON

避免过度依赖：

- 只用 `aiAssert("页面正常")` 这种模糊判断

推荐模式：

1. `aiAssert`：确认页面不是登录页/报错页/空白页
2. `aiQuery`：提取结构化指标
3. JavaScript：执行阈值、波动、缺失、组合规则判断

### 6.4 YAML 的定位

本产品中 YAML 不是主执行形态，而是辅助能力：

- 可用于录制结果导入
- 可用于调试和复现场景
- 可作为高级用户的任务描述格式

但主产品体验应以 **结构化任务配置 + JS 编排执行** 为主。

不要将整个产品做成“YAML 文件管理器”。

## 7. 告警设计原则

- Electron 弹窗通知
- 可选声音提醒

### 7.2 必须做去重和恢复

后续实现不要只做“每次失败就发一条”。

最低要求：

- 首次异常通知
- 持续异常抑制/合并
- 恢复通知

### 7.3 告警内容

每条告警建议至少包含：

- 任务名称
- 目标系统
- 异常类型
- 异常指标值
- 阈值/规则
- 执行时间
- 最近截图
- 报告入口

---

## 8. 调度设计原则

### 8.1 调度模式

推荐采用：

- Electron 主进程托盘常驻
- 内置 scheduler 定时触发任务

不要把主业务逻辑放到一次性命令行进程里。

### 8.2 调度控制

每个任务建议支持：

- 执行间隔（如每 10 分钟）
- 生效时间段
- 默认启动立即执行
- 超时限制
- 失败重试次数

### 8.3 并发策略

默认应以 **串行执行** 为主。

理由：

- Bridge 接管的是同一个桌面 Chrome 会话
- 并发执行多个任务容易互相污染状态
- 对巡检产品而言，稳定性优先于吞吐量

只有在后续明确实现“多浏览器实例 / 多机器分片”后，才考虑并发。

## 9. UI 规划

### 9.1 总体原则

UI 应设计成 **工作台**，不是聊天框，不是脚本 IDE，也不是浏览器。

目标体验应偏向：

- 控制台
- 配置台
- 告警与报告记录存档（只本地存储即可）

### 9.2 导航结构

建议采用左侧导航，至少包含：

1. **总览**
2. **任务**
3. **执行记录**
4. **告警中心**
5. **模型与连接设置**
6. **系统设置**

### 9.3 总览页

建议展示：

- 当前 Bridge 状态
- Chrome 扩展连接状态
- 今日执行次数
- 异常任务数
- 下次计划执行时间

### 9.4 任务列表页

每个任务卡片/行建议包含：

- 名称
- 系统/分组
- 状态（正常 / 异常 / 登录失效 / 暂停）
- 最近一次结果
- 上次执行时间
- 下次执行时间
- 快捷操作（运行、暂停、编辑、查看报告）

### 9.5 创建任务向导

建议分 5 步：

#### 第一步：连接与运行方式

- 选择 Bridge 运行方式：
  - `newTabWithUrl`（推荐）
  - `currentTab`（调试）
- 检测扩展连接状态

#### 第二步：目标页面

- 任务名称
- 目标系统
- 入口 URL
- 页面说明

#### 第三步：页面就绪与取数

- 页面就绪提示词
- 结构化提取提示词
- 返回 schema 编辑器
- 测试执行一次

#### 第四步：异常规则

- 阈值规则
- 波动规则
- 缺失规则
- 登录失效规则
- 高级表达式

#### 第五步：调度与告警

- 执行频率
- 工作时段
- 重试策略

### 9.6 记录页

每次运行至少展示：

- 开始/结束时间
- 状态
- 耗时
- 提取结果 JSON
- 命中规则
- 截图
- 报告入口
- 原始日志

### 9.7 告警中心

不要只做一个平铺列表，建议具备事件生命周期：

- 未处理
- 已确认
- 已静默
- 已恢复

### 9.8 模型设置页

建议 UI 设计为：

- **默认执行模型**（默认展开，必填）
- **高级模型策略**（折叠面板）
  - 独立页面理解模型
  - 独立任务规划模型

## 10. 配置与数据存储建议

### 10.1 配置分层

建议分为三层：

1. **应用级配置**
   - Bridge 端口
   - 扩展状态
   - 模型配置
   - 日志级别
   - 自动启动设置

2. **任务级配置**
   - URL
   - 提示词
   - schema
   - 调度
   - 告警

3. **运行态数据**
   - 最近结果
   - 历史记录
   - 告警状态
   - 报告索引

### 10.2 数据落地

优先建议均使用JSON 配置 + 文件目录，目前无多任务、多历史需求

## 11. 对后续 AI Coding 的明确要求

### 11.1 优先事项

按优先级实现：

1. Bridge-only 连接闭环
2. 任务模型与调度器
3. `aiQuery + 规则判断` 的巡检执行链
4. 告警中心
5. 报告与运行历史
6. 模型高级配置
7. 录制/YAML 导入增强

### 11.2 不要做的事情

后续编码时禁止：

- 重新引入 CDP/Launch 选项到 UI
- 默认并发执行多个 Bridge 任务
- 把产品做成聊天式 playground
- 先做复杂 YAML 编辑器而忽略任务配置和告警
- 假设规划模型是纯文本模型
- 只用 `aiAssert` 做所有数值判定

### 11.3 监控任务的推荐技术策略

对于巡检任务，优先采用以下模板：

1. 连接 Bridge
2. 打开目标页
3. `aiWaitFor` 页面就绪
4. `aiAssert` 检查非登录页/非错误页
5. `aiQuery<T>` 提取结构化数据
6. JavaScript 规则引擎判断异常
7. 保存截图、日志、报告
8. 发告警

## 12. 里程碑建议

### M1：Bridge 专用化闭环

- 移除非 Bridge 用户入口
- 配置结构收敛
- 能手动执行一次任务

### M2：单任务巡检闭环

- 可创建一个任务
- 可按固定频率执行
- 可提取数据并判断异常
- 可本地通知

### M3：工作台成型

- 任务列表
- 执行记录
- 告警中心
- 设置页

### M4：高级能力

- 高级模型策略
- 任务导入导出
- Recorder / YAML 集成
- 更完善的异常抑制与恢复通知

---

## 14. 最终产品决策摘要

如后续实现遇到取舍不清，请以本节为准：

1. **只做 Bridge，不做 CDP/Launch**
2. **Electron 是工作台，不是浏览器**
3. **默认面向专用值守电脑**
4. **定时任务优先 `newTabWithUrl`，调试可 `currentTab`**
5. **监控巡检优先 `aiQuery + 规则引擎`**
6. **默认视觉模型必填**
7. **Planning/Insight 为可选高级配置**
8. **如果只能做两套模型配置，优先 默认模型 + Insight**
9. **不要把 Planning 设计成纯文本模型**
10. **先把调度、告警、报告做扎实，再做录制/YAML 增强**
